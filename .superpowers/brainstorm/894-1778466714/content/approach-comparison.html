FSM Architecture Approaches

3 ways to organize the XState Actor network for K8s simulation

A

Peer Actor Network

No central coordinator. Each component (api-server, etcd, scheduler, kubelet...) is an independent actor. Components send messages directly to each other, just like real K8s. The simulation phase emerges from message flow.

Pros

  • Most faithful to real K8s architecture
  • Components are truly independent and testable
  • Easy to add new components (just add an actor)
  • Naturally supports concurrent message flows

Cons

  • Hardest to control playback (step-forward/backward)
  • Testing non-deterministic message ordering
  • No single source of truth for "what phase are we in"
B

Hub-and-Spoke with API-Server as Hub

API-Server actor acts as the message hub (like real K8s). All component communication goes through it. It routes messages and tracks phases. Other components (etcd, scheduler, kubelet) are spoke actors.

Pros

  • Mirrors real K8s API-Server as the communication hub
  • API-Server can track phase progression
  • Better playback control (messages flow through one point)
  • Clear message ordering and causality

Cons

  • API-Server actor becomes complex (routing + state)
  • Not all K8s communication goes through API-Server (e.g., kubelet→CRI)
  • Slightly artificial for plugin-to-plugin messages
C

Orchestrator + Component Actors

A simulation orchestrator actor manages the overall flow (phases, playback). Component actors handle internal state (idle→processing→done/error). The orchestrator sends commands and collects results, components respond with events.

Pros

  • Best playback control — orchestrator drives step-by-step
  • Clear separation: orchestration vs component logic
  • Easy to implement pause/resume/jump-to
  • Phase transitions are explicit and testable

Cons

  • Less "real" — real K8s has no orchestrator
  • Orchestrator can become a god object if not careful
  • Need careful boundary between orchestrator and component responsibilities

Recommendation: Approach C

The orchestrator pattern gives the best balance for a visualization tool. Real K8s fidelity (Approach A) is less important than playback control and user experience. The orchestrator naturally maps to the existing timeline/playback UI.
