Actor FSM Definitions — Part 2: Scheduling & Control

SchedulerActor & ControllerManagerActor

SchedulerActor
States:
  idle · filtering · scoring · binding
  error

Transitions:

idle ──[POD_PENDING]──→ filtering
  │ load pod spec, start filter
  │
  ├─[FILTER_DONE, candidates > 0]──→ scoring
  │   run score plugins on candidates
  │
  ├─[FILTER_DONE, candidates = 0]──→ idle
  │   emit POD_UNSCHEDULABLE
  │
  ├─[FILTER_ERROR]──→ error
  │   emit POD_UNSCHEDULABLE
  │
  ▼
scoring ──[SCORE_DONE]──→ binding
  │
  ├─[SCORE_DONE, all scores = 0]──→ idle
  │   emit POD_UNSCHEDULABLE
  │
  ▼
binding ──[BIND_OK]──→ idle
  │ emit BIND_REQUEST → APIServer
  │
binding ──[BIND_FAIL]──→ idle
  │ requeue pod (publish POD_PENDING)
  │
binding ──[TICK]──→ binding
  │ retry timeout, emit BIND_FAIL

Error Recovery:
error ──[TICK]──→ idle
  │ backoff, requeue pod

Context Data

pendingPod Pod currently being scheduled
candidates Nodes passing filter phase
scores Map<nodeName, score>
filterPlugins Extensible filter chain
ControllerManagerActor
States:
  watching · reconciling · updating
  error

Transitions:

watching ──[POD_CREATED]──→ reconciling
watching ──[POD_UPDATED]──→ reconciling
watching ──[POD_FAILED]──→ reconciling
  │
  ▼
reconciling ──[diff > 0]──→ updating
  │ calculate desired vs actual
  │ for each managed controller:
  │   compute diff
  │   generate CREATE/DELETE actions
  │
reconciling ──[diff = 0]──→ watching
  │ nothing to do
  │
  ▼
updating ──[UPDATE_DONE]──→ watching
  │ publish CREATE_POD / DELETE_POD
  │ for each diff action
  │
updating ──[UPDATE_FAIL]──→ error
  │
Operator Chain (BFS):
watching ──[CRD_ADDED]──→ reconciling
  │ dispatch to matching Operator
  │ new resources trigger next
  │ controller in chain (depth ≤ 3)
  │
  │ e.g. Deployment → ReplicaSet
  │      → Pod (chain reconcile)
  │
Error Recovery:
error ──[TICK]──→ watching
  │ backoff, re-watch

Managed Controllers

Each built-in controller (Deployment, ReplicaSet, DaemonSet, Job, CronJob) is a strategy object inside ControllerManagerActor, not a separate Actor.

This mirrors real K8s: one controller-manager process runs multiple controller loops.

Custom operators from PluginEditor become additional strategy objects registered at runtime.
