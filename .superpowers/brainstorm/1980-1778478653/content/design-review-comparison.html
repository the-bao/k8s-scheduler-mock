Design Gap Analysis

Our approach vs. the k8s-fsm-engine-design document

What the document adds that we're missing

1

5-Layer Architecture

The document defines L1 (UserAction + TickClock) → L2 (MessageBus) → L3 (Actor Runtime) → L4 (K8s Domain) → L5 (View). Our 3-layer design (Actors + EventBus + Playback) is less granular.

2

TickClock

Explicit time driver for health probes, retries, and periodic reconciles. We don't have this — our auto-play timer only controls playback speed, not simulated K8s time.

3

EtcdStore (Simulated etcd)

Dedicated in-memory etcd layer with version numbers. Currently our Zustand store conflates UI state and K8s resource state. Separating them is cleaner.

4

EventHistory with Deterministic Replay

Instead of snapshots, the document replays from initial state using recorded event sequence. More memory-efficient for long simulations. Same events → same state (determinism guarantee).

5

Formal Actor Base Class with Mailbox

Each Actor has a mailbox queue with sequential draining. Prevents re-entrant processing. XState actors handle this internally but less transparently.

6

ActorRegistry

Lookup actors by ID, support multiple instances of same type (e.g., KubeletActor per node). We didn't design this explicitly.

7

Declarative Transition Table with Guards

Plain array of {from, event, to, guard?, action?} rules. Extremely readable. XState uses a different DSL (object-based) which is more verbose for simple cases.

8

ReactiveStore (useSyncExternalStore)

Direct React 18 integration without Zustand dependency. Simpler, standard React API. We planned to keep Zustand as the view state bridge.

Key Question

The document's custom FSMEngine is ~20 lines of code. It's purpose-built, zero-dependency, and does exactly what this project needs. Meanwhile, XState is a full-featured library (32KB min) with its own concepts and learning curve.

Should we reconsider our XState choice?
